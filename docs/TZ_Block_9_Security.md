## 3. Аутентификация

### Пароли
- **Хранение:** `.env` файл (жёстко заданы)
- **Формат:** `PASSWORD_ROLE=хеш_или_пlaintext_в_первой_версии`
- **Смена:** Только через деплой (перезапуск бота)

```bash

PASSWORD_AGENT=agent2025
PASSWORD_MANAGER=manager2025
```

### Сессия
- **Срок:** 8 часов с момента ввода пароля
- **Продление:** Автоматически при активности
- **Повторная авторизация:** Только при истечении сессии
- **Хранение:** В памяти бота (dict: `user_id → {role, expires_at}`)

- После 5 неудачных попыток — блокировка на 15 минут

## 4. Шифрование данных

### Рекомендация для MVP: **SQLCipher** (вариант B)

**Почему не D (отдельные поля):**
- Сложнее реализовать (ручное шифрование/дешифрование в коде)
- Риск ошибки (забыть расшифровать при выводе)
- Проблемы с поиском (нельзя искать по зашифрованному полю)

**SQLCipher — проще:**
- Шифрование всей БД одним ключом
- Прозрачно для приложения (работа как с обычной SQLite)
- Один параметр при подключении: `PRAGMA key = 'пароль'`

### Реализация
```python
# database.py
import sqlite3
from pysqlcipher3 import dbapi2 as sqlcipher  # SQLCipher для Python

def get_db_connection():
    conn = sqlcipher.connect('/root/projects/py/data/database.db')
    conn.execute(f"PRAGMA key = '{os.getenv('DB_ENCRYPTION_KEY')}'")
    return conn
```

**Ключ в `.env`:**
```bash
DB_ENCRYPTION_KEY=YourStrongPassword123!
```

### Что шифруется
- Вся база данных (включая логи, пароли пользователей бота, телефоны клиентов)
- Резервные копии на Яндекс.Диск (уже шифруются в Блоке 8)

## 5. Логирование безопасности

### Фиксируемые события (все)

| Событие | Уровень | Действие при подозрении |
|---------|---------|------------------------|
| Вход в бота (успешный) | INFO | — |
| Вход в бота (неудачный) | WARNING | Счётчик попыток |
| Просмотр чужой сметы | ERROR | Алерт в Telegram-лог |
| Создание сметы | INFO | — |
| Изменение сметы | INFO | — |
| Финализация сметы | INFO | — |
| Генерация документов | INFO | — |
| Внесение расхода | INFO | — |
| Изменение чужого расхода (CEO) | WARNING | Лог + уведомление автору |
| Экспорт аналитики | INFO | — |
| Попытка доступа к запрещённому | ERROR | Блокировка + алерт |
| Истечение сессии | INFO | — |

### Хранение логов
- **Основное:** Таблица `security_logs` в SQLCipher-БД
- **Копия:** Яндекс.Диск (ежедневно с задержкой 24 часа)
- **Критичные:** Telegram-лог чат (мгновенно)

### Структура таблицы `security_logs`

```sql
CREATE TABLE security_logs (
    id INTEGER PRIMARY KEY,
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    user_id INTEGER,
    user_role TEXT,
    action TEXT,           -- 'login', 'view_estimate', 'edit_expense'...
    object_type TEXT,      -- 'estimate', 'expense', 'client_data'...
    object_id INTEGER,     -- ID сущности
    result TEXT,           -- 'success', 'denied', 'error'
    ip_address TEXT,       -- если доступно
    details TEXT,          -- JSON с доп. данными
    alert_sent BOOLEAN DEFAULT 0  -- был ли отправлен алерт
);
```

## 11. Что НЕ реализуется (в MVP)

| Функция | Причина | Когда |
|---------|---------|-------|
| 2FA/TOTP | Пароля достаточно для MVP | При расширении команды |
| Хеширование паролей в `.env` | Удобство смены, VPS защищён | v2 |
| Шифрование отдельных полей | SQLCipher проще | Не требуется |
| Автоматическая смена паролей | Ручное управление надёжнее | При утечке |
| Внешний пентест | Нет ресурсов | Перед масштабированием |
| CloudFlare/DDoS-защита | Telegram API защищает | При webhook + высокой нагрузке |
